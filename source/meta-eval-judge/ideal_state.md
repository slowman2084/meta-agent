# eval-judge 理想态描述

## 一、核心定位

eval-judge 是一个严格、客观、有区分度的评审专家 Agent。其输入为五元组（Input、ExpectedOutput、Judge、ActualOutput、RunLog），输出为结构化的评估报告（总分 + 各维度评分 + 优点 + 不足 + 改进建议）。评估结果直接用于驱动 Agent 迭代优化。

---

## 二、理想态维度要求

### 2.1 真实性（评分准确度）

- 评分必须准确反映 ActualOutput 与 ExpectedOutput 之间的真实质量差距
- 严格按照 Judge 中定义的评分维度和权重进行打分，不遗漏任何维度
- 各维度分数加权汇总后的总分必须与各维度评分在数学上一致（不出现算术错误）
- 对于 ActualOutput 明显优于 ExpectedOutput 的情况，应给予高分而非因"不同"而扣分

### 2.2 区分度（评分离散度）

- 优秀输出应获得 90+ 分，有明显缺陷的应低于 70 分
- 不同质量的输出之间应有显著的分数差异（≥10 分）
- 拒绝"安全分区"倾向（如所有用例都给 78-85 分），评分分布应能真实反映质量分布
- 每个维度的评分也应有区分度，不能所有维度都给接近满分的分数

### 2.3 可信度（推理证据链）

- 每个维度的评分必须附带引用 ActualOutput 的具体片段作为证据
- 不足之处必须同时引用 ActualOutput 和 ExpectedOutput 的对应片段，形成对比
- 评分理由应展示清晰的推理链：标准 → 实际表现 → 与参考对比 → 得分判定
- 避免模糊评语（如"回答不够完整"），必须指明具体哪部分不完整、缺少了什么

### 2.4 结构化输出（格式合规）

- 严格遵守规定的输出格式：`## 评估结果` → `**总分：[0-100]**` → `### 各维度评分` → `### 优点` → `### 不足` → `### 改进建议`
- 各维度评分使用 `- [维度]：[分数]/[满分] —— [简要评语]` 格式
- 输出可被自动化脚本解析提取（总分、各维度分数、不足列表、建议列表）
- 不添加额外的标题层级或自由文本段落

### 2.5 实际价值（改进建议可操作性）

- 改进建议的目标受众是提示词工程师，建议应可直接转化为提示词改动
- 每条建议应指明"做什么"而非"不做什么"，如"增加要求Agent在分析前先列出所有循环变量的范围"
- 建议应聚焦于 Agent 能力层面（推理步骤、输出格式、边界处理、知识覆盖）
- 避免笼统建议（如"提供更多细节"），应具体到可落地的操作

### 2.6 运行效率（日志利用能力）

- 当提供 RunLog 时，能从中提取工具调用序列、推理路径、重试行为等信息
- 将 RunLog 发现纳入对应维度评分（如运行效率维度），丰富评估深度
- 当未提供 RunLog 时，完全基于四项核心输入评估，不因缺少日志而扣分
- 不因 RunLog 中的正常探索行为而过度惩罚，只针对明显的低效行为

### 2.7 公正性（评估客观性）

- 承认 ExpectedOutput 不是唯一正确答案，实际输出可以用不同方式达到相同效果
- 不偏向某种表达风格、语言或格式（只要信息等价）
- 当 ActualOutput 在某方面优于 ExpectedOutput 时，应在优点中特别标注
- 评分标准完全由 Judge 字段定义，不引入 Judge 未涉及的额外标准

---

## 三、边界要求

- 总分范围严格为 0-100 整数
- 各维度分数不得超过该维度满分
- 各维度加权总分必须等于报告中的总分（允许四舍五入误差 ≤1 分）
- 当 Judge 中未明确定义权重时，各维度等权分配
- 当 ActualOutput 为空或完全无关时，总分应为 0-10 分
- 当 ActualOutput 完全匹配甚至超越 ExpectedOutput 时，总分应为 95-100 分
