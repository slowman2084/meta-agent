# 歌词生成系统 迭代优化变更记录

## 第三轮优化（iter_3, 本轮优化）

**问题诊断：**
1. **金句原创性不足**（Case 1）：大量沿用参考范例原句，虽然提示词已有“不可抄袭”的目标要求，但缺乏执行路径，导致模型无法独立创作。
2. **副歌递进层次缺失**（Case 3）：副歌核心句重复时完全相同，无情感递进的细微变化。
3. **生成后自检缺失/逻辑错误**（Case 4）：副歌出现了"有关"与"无关"前后矛盾的逻辑错误。

**内/外因判断：**
这三个问题均属于**内因**（Agent 自身推理缺陷）。原提示词堆砌了近千行的规则（客体视角的目标），但缺乏让 Agent 进行前置规划和检查的推理空间（主体视角的行动），导致 Agent “知其然而不知如何做到”。

**四因归因：**
- **金句原创性不足**：动力因缺失。Agent 看到了"要原创"的指令，但没有执行"拆解范例结构 -> 替换核心词汇"的强制推理步骤。
- **副歌递进缺失**：动力因缺失。缺乏在生成正文前，进行多版本副歌对比规划的机制。
- **逻辑一致性错误**：形式因与动力因缺失。Agent 边想边写，容易导致前后文"定调词"自相矛盾，且缺乏硬性的防偏离逻辑基准。

**CoT 抽象结论：**
规则堆砌无法代替执行步骤。Agent 需要一个强制的 CoT 工作流（`<thinking>`）来前置消化所有硬性约束：
- 要原创，就必须在规划阶段强制明确宣告"保留的结构"和"替换的新词汇"。
- 要递进，就必须在规划阶段预先写出 Chorus 1/2/3 的对比版本。
- 要防逻辑矛盾，就必须在提笔前划定"逻辑红线"（如：全篇死守"表面无关"基调，一旦出现"真正有关"即为违规跳点）。

**优化策略：**
1. **引入强制的 CoT 推理框架**：重构输出模板，强制 Agent 首先输出 `<thinking>` 标签，规划关键卡点后再生成歌词。
2. **将后置要求转为前置规划**：在思考区新增"金句原创与副歌递进规划"板块，明确规划 Chorus 1/2/3 的递进词汇。
3. **建立防偏离逻辑基准**：在思考区新增"情感定调与逻辑基准"，确立全篇逻辑红线。
4. **化繁为简**：将原提示词中冗长散乱的规则，系统梳理提炼为 MECE 的四大核心能力区，强效聚焦 Agent 注意力。

**改动范围：**
- 大幅重构提示词结构，从"堆砌大量防错规则"转向"标准工作流驱动"。
- 新增 `标准工作流（CoT 推理框架）` 章节。
- 优化了输出格式规范和生成后自检报告的精确度要求，做到自检项与推理强绑定。

**本轮优化方向标签：** CoT框架 / 输出格式 / 逻辑统一
